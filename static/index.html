A note for the bad days

You're not behind — you're over-prepared in the wrong currency. Thirteen merged PRs in a Rust Foundation linker. A published MoE framework. Real systems depth. For you, DSA is the one missing key to a door you've already built the house behind.

This isn't a betrayal of being a builder. It's a toll. Pay it fast, pay it ugly, in a language you can stand — then get back to the work that's actually yours.

DSA gets you past the OA. CS fundamentals get you through the interview. OAs are ~90% DSA plus a few MCQs; these subjects decide the technical rounds after. So this is a second-phase priority — heavy from Jul 13 onward, not competing with the sprint now. The only fundamentals to touch before then are the OOP/LLD day already in Sprint 2.

Tiers:  CORE asked in almost every interview regardless of role — non-negotiable.  ·  ROLE matters specifically for ML/AI or systems/AppSec — your differentiation.  ·  THIN occasional MCQs — know headlines, don't go deep.

OOP & Design
Highest frequency · lowest effort · asked everywhere
CORE ▾
Core — must own
  • The four pillars — encapsulation, abstraction, inheritance, polymorphism — each with a real code example, not a textbook definition.
  • Composition vs inheritance, and why "favor composition." You live this in Go, which has no inheritance — a strong honest answer.
  • SOLID — each principle with a one-line "what breaks if you violate it."
  • Abstract class vs interface; when to use each.
  • Static vs dynamic (runtime) polymorphism; overloading vs overriding.
Core design patterns — the 6 that get asked
  • Singleton (and why it's often an anti-pattern — good senior signal).
  • Factory / Factory Method.
  • Observer. · Strategy. · Decorator. · Adapter.
Role — ML / systems
  • How OOP maps to Go's composition + interfaces (defend Go's lack of inheritance — easy win).
  • Dependency injection in practice.
Thin
  • Remaining GoF patterns — recognize the names, don't memorize.
Prep move
  • Already a line item in DSA Sprint 2. One focused day. Resource: refactoring.guru for patterns.

DBMS
Double-counts — OA MCQs/SQL + interview concepts · for ML this beats networking
CORE ▾
Core — must own
  • Normalization — 1NF→BCNF, and actually normalize a given table, not recite. The single most-asked DBMS topic. Trade-off: normalize for integrity, denormalize for read speed.
  • SQL fluency — joins (inner/outer/self), GROUP BY + HAVING, subqueries, window functions, RANK/DENSE_RANK/ROW_NUMBER. Drillable like DSA — practice on LeetCode SQL / HackerRank.
  • Indexing — B-tree vs hash, clustered vs non-clustered, why an index speeds reads & slows writes, when it's useless (low cardinality, leading-wildcard LIKE).
  • Transactions & ACID — each letter with a concrete failure it prevents.
  • Isolation levels — read-uncommitted → serializable, and the anomalies each allows (dirty / non-repeatable / phantom reads).
  • Joins under the hood — nested-loop vs hash vs merge join.
  • ER modeling — entities, relationships, cardinality; word problem → schema.
Role — ML track
  • SQL vs NoSQL trade-offs; when document / columnar / key-value wins.
  • OLAP vs OLTP, warehouse vs lake (ML data pipelines).
  • CAP theorem — pick-two framing and why it's a simplification.
Thin
  • Relational algebra notation; deep query-optimizer internals.
Prep move
  • SQL = mini-DSA drill. Normalization + transactions + indexing = the conceptual core. ~3–4 focused days.

Operating Systems
Universal · ROLE-heavy for systems/AppSec · you have a Silberschatz base
CORE ▾
Core — must own
  • Process vs thread — address space, what's shared, context-switch cost. The most-asked OS question, period.
  • Concurrency & synchronization — race conditions, critical section, mutex vs semaphore, monitors. Spot a race in code.
  • Deadlock — four Coffman conditions, prevention vs avoidance vs detection, Banker's algorithm conceptually.
  • CPU scheduling — FCFS, SJF, Round Robin, priority, MLFQ; compute avg waiting/turnaround on a workload (common numeric).
  • Memory management — paging, segmentation, virtual memory, page faults, TLB.
  • Page replacement — FIFO, LRU, Optimal; Belady's anomaly. Often "how many page faults for this reference string?"
  • Classic problems — producer-consumer, readers-writers, dining philosophers.
Role — systems / AppSec
  • System calls, user vs kernel mode, mode switch vs context switch.
  • Process memory layout (stack/heap/data/text) — directly relevant to your security / CTF / exploitation work.
  • File systems — inodes, path resolution (your Linux depth helps).
  • fork / exec / wait, zombie & orphan processes, copy-on-write.
Thin
  • Disk scheduling (SCAN / C-SCAN) — names only; RAID levels.
Prep move
  • You have a real base. Refresh scheduling + page-replacement numerics (free points), make concurrency explanations crisp. ~3 days.

System Design & ML System Design
The real differentiator for your primary track — where most candidates are weak
ROLE ▾
Core vocabulary — everyone needs
  • Scalability (vertical vs horizontal), load balancing, caching (where, eviction, invalidation).
  • DB sharding & replication, read replicas, leader-follower.
  • CAP theorem & consistency models (strong vs eventual).
  • Message queues / pub-sub, async decoupling.
  • API design — REST, idempotency, rate limiting.
  • Back-of-envelope estimation (QPS, storage, bandwidth).
  • Warm-ups: design a URL shortener, rate limiter, pastebin.
ML System Design — your edge, prioritize
  • The flow interviewers want: clarify → frame as ML → data → features → model → training → evaluation → serving → monitoring.
  • Framing a business problem as an ML problem (heavily tested).
  • Data collection, labeling, train/val/test splits, leakage.
  • Feature engineering & feature stores.
  • Offline vs online metrics; how you'd measure success.
  • Serving — batch vs real-time inference, latency/cost/accuracy trade-offs.
  • Production realities — concept drift, monitoring, feedback loops, retraining, A/B tests. Your DL_MoE parameter-efficiency work is a genuine asset here.
  • Practice prompts: recommendation system, feed ranking, fraud detection, search ranking, ETA.
Prep move
  • This is a talking skill — practice out loud with a framework (RESHADED for general; clarify→frame→data→model→serve→monitor for ML). Watch 3–4 worked ML-design videos, then mock. Late-stage, high leverage.

Computer Networks
A few near-guaranteed Qs · goes DEEP for AppSec
CORE-lite ▾
Core — the near-guaranteed ones
  • OSI vs TCP/IP — layers and what each does. Low effort, frequent.
  • "What happens when you type a URL and press enter?" — the most famous systems/networking question. DNS → TCP handshake → TLS → HTTP → response → render. Know it cold; it touches everything.
  • TCP vs UDP — guarantees, when each, the 3-way handshake.
  • HTTP/HTTPS — methods, status codes, statelessness, how TLS works at a high level.
  • DNS — resolution flow, record types. IP addressing, subnetting basics, NAT.
Role — AppSec (gets deep)
  • TLS handshake in detail, certificates, PKI.
  • Web attacks & defenses — XSS, CSRF, SQLi, SSRF (overlaps your PortSwigger / THM work directly).
  • Same-origin policy, CORS, cookies/sessions, JWT.
  • Security headers (CSP, HSTS); firewalls, ports, proxy/load-balancer placement.
Thin (for your roles)
  • Routing algorithms (distance-vector vs link-state), congestion-control math, ARP details.
Prep move
  • Nail the URL→render walk + OSI/TCP-UDP/HTTP (~2 days). Your AppSec study doubles as networking interview prep.
